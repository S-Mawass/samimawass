Why this project

Most portfolios are templates filled in and forgotten. I treated this one as a product: defined the audience (GCC recruiters and consulting clients), set quality criteria up front — WCAG 2.1 AA, full SEO metadata, a design system that holds across 8 pages — and only then started building. Positioning decisions came first; Claude Code implemented them; every file was reviewed before commit.

Technical decisions

  • No frameworks, no build step — pure HTML/CSS/JS keeps the codebase portable and forces every design decision to be explicit rather than inherited from defaults.
  • Design tokens via CSS custom properties — Azure Blue and Fabric Green (a deliberate nod to the Microsoft ecosystem my audience works in), two typefaces, no hardcoded hex values anywhere.
  • Responsive at 600px and 900px with Grid and Flexbox only.
  • JavaScript limited to two concerns — scroll-reveal (IntersectionObserver) and mobile nav.
  • Accessibility built in, not retrofitted — skip links, ARIA landmarks, focus-visible on every interactive element.
  • SEO from day one — canonical URLs, sitemap, Open Graph/Twitter Cards on every page, JSON-LD Person schema.

Outcomes

  • A coherent design system maintained across 8 pages with custom properties alone — no visual drift.
  • 8 client projects published as anonymised case studies with zero client names anywhere in the codebase, including comments and filenames.
  • Fully portable: no dependencies, no node_modules, nothing to maintain but the content.

Lessons

  • AI-assisted development rewards clear briefs more than clever prompts — vague prompts produce vague code.
  • The review step is not optional: every generated file was read before commit, which caught accessibility gaps a copy-paste workflow would ship.
  • Constraints clarify: banning frameworks made every decision deliberate.
