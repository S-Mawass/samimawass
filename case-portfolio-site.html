Project Overview

Most professionals treat a portfolio website as a chore — a template filled in and forgotten. This project treated it as a product: something with a defined audience, a positioning strategy, a design system, measurable quality criteria, and a continuous deployment pipeline. The result is the site you are reading now — built entirely without a framework, without a CMS, and without outsourcing any part of the decision-making to a template provider. The entire process, from blank canvas to live production, was executed using an AI-native workflow across six tools.

Challenge

The challenge was not purely technical. Any developer can put together a static site. The harder problem was positioning: who is this site for, what should a visitor believe after reading it, and how should the architecture and content hierarchy reflect that intent. At the same time, the technical execution had to meet the same standards I hold client deliverables to — WCAG 2.1 AA accessibility, structured SEO metadata, responsive breakpoints at 360/768/1200px, and a design system coherent enough to scale to 20+ pages without visual drift.

Objectives

  • Define a clear positioning and audience — analytics engineering, BI development, UAE/GCC market — and express it through every content and design decision.
  • Build a reusable design system (tokens for colour, type, spacing, and motion) that would hold together across 21+ pages.
  • Implement 18 anonymised client case studies, each with consistent structure, no invented metrics, and no client names anywhere in the codebase.
  • Meet WCAG 2.1 AA with skip links, ARIA labels, focus-visible outlines, and semantic HTML throughout.
  • Deploy to GitHub Pages with a clean canonical URL structure, sitemap, and Open Graph / Twitter Card metadata on every page.

Solution Approach

Rather than starting with code, the process started with positioning. What does an Analytics Engineer with a Power BI and SSAS Tabular background need a recruiter or client to understand within 30 seconds? That question drove content hierarchy before a single line of HTML was written. The design system — a set of CSS custom properties for a Microsoft-adjacent palette, two typefaces, and a consistent component vocabulary — was designed before being handed off to Claude Code for implementation. Every iteration followed the same pattern: decision first, implementation second, review third.

Design & Branding Process

The design system was built around a Microsoft-adjacent palette — Azure Blue (`#0078D4`) and Fabric Green (`#15B392`) as primary and accent, set against a dark navy ink surface and a near-white foreground. This wasn't arbitrary: the portfolio's audience includes people who work in the Microsoft Power Platform ecosystem, and the visual language signals fluency without stating it. Two typefaces were chosen for their contrast and complementarity: Space Grotesk for headings (geometric, confident, modern) and Inter for body text (neutral, highly legible at small sizes). IBM Plex Mono was added for code and metadata — a subtle signal of technical credibility. All values were expressed as CSS custom properties (`--azure`, `--fabric`, `--head`, `--body`, `--mono`) so they could be applied consistently without ever hardcoding a hex value in a component.

Development Process

The build followed a strict constraint: no external frameworks. No Tailwind, no Bootstrap, no React, no build step. Every class was written by hand in a single `style.css` file. This was a deliberate choice — a framework would have solved layout problems faster but introduced a dependency that would require maintenance, upgrades, and familiarity from any future contributor. Instead, CSS custom properties, Grid, and Flexbox were used throughout, with a `@media` breakpoint system targeting 600px, 900px, and 1200px. JavaScript was limited to two concerns: scroll-reveal animations via IntersectionObserver, and the mobile navigation toggle. Accessibility was built-in from the start — not retrofitted — with skip links, `id="main-content"` landmarks, `aria-current="page"` on active nav items, and `focus-visible` outlines on all interactive elements.

Deployment & Version Control

Every change was committed atomically with a descriptive message, producing a commit history that reads as a project diary. Deployment is handled entirely by GitHub Pages — a push to `main` triggers an automatic build and deploy in under 30 seconds. The site is served from a clean canonical URL (`https://sami-mawass.github.io/sami-mawass/`) with a `sitemap.xml` listing all 22 URLs and `robots.txt` pointing crawlers to it. Every page carries canonical link tags, Open Graph metadata, and Twitter Card tags. JSON-LD Person schema is embedded in the homepage for structured data.

Key Outcomes

  • ~70% development time reduction — AI-assisted code generation, content drafting, and design iteration compressed a project that would typically take weeks of solo effort into focused sessions.
  • 21+ pages shipped — homepage, about, client projects, personal projects, and 18 individual case study pages, all from a single coherent design system.
  • Zero framework dependencies — the codebase is fully portable, readable without tooling, and has no node_modules, no build step, and no lock file to maintain.
  • WCAG 2.1 AA — accessibility built into the architecture, not patched on afterwards.
  • Full SEO coverage — canonical URLs, Open Graph, Twitter Card, JSON-LD schema, sitemap, and robots.txt on day one.
  • Design system coherence at scale — 21+ pages with consistent visual language, spacing, and component vocabulary maintained through CSS custom properties alone.

Lessons Learned

The most valuable lesson was that AI-assisted development rewards clear thinking more than it rewards prompt cleverness. The better the brief — audience, constraints, tone, structure — the better the output. Vague prompts produce vague code. The second lesson was that the review step is not optional: AI-generated code should be read, not just accepted. Every file in this project was reviewed before committing, which caught edge cases, accessibility gaps, and inconsistencies that would have gone unnoticed in a pure copy-paste workflow. The third lesson was about constraint: limiting the technical stack to HTML/CSS/JS forced every design decision to be explicit rather than inherited from a framework's defaults.
