Problem

Opaque reporting with no semantic layer — and a 30-minute refresh.

A national higher-education institution in the UAE was managing institutional planning through fragmented sources and manually produced reports. There was no unified semantic layer: each report carried its own logic, measure definitions drifted across files, and no documentation existed to help stakeholders verify what they were looking at.

Dataset refresh took approximately 30 minutes — long enough that planning meetings often ran on stale figures. Row-level security was managed inconsistently, with no formal handoff to the internal team.

The core issue wasn't missing data — it was a model architecture that hadn't been designed to support scale, clarity, or trust.

Context

On-premise stack, multi-role access, accreditation requirements.

The institution operated on an on-premise Microsoft stack: SQL Server as the source system, with Power BI Report Server for distribution. Reporting needed to serve multiple user roles — institutional planners, faculty leads, and executive stakeholders — each with different data access requirements.

Reports fed into accreditation workflows, where accuracy and consistency were non-negotiable. The solution needed to be auditable, maintainable, and owned by the internal team after handoff.

Approach

Audit, re-architect, document, hand off.

  • Audit the existing model — map the current data grain, identify repeated logic and conflicting measure definitions, document what was implicit.
  • Re-architect into a star schema — separate fact and dimension tables cleanly; eliminate the implicit many-to-many relationships causing slow refresh and inconsistent aggregations.
  • Build the SSAS Tabular semantic model — define a shared DAX measure library, standardise KPI naming, and implement calculation groups for time intelligence.
  • Design and implement production RLS — role-based row-level security scoped to institutional hierarchy, with written documentation enabling the internal team to manage access going forward.
  • Deliver Power BI reports — live connection from Report Server to the SSAS model, scoped to the final agreed KPI set.
  • Train the internal owner — a business user was trained to manage RLS independently, reducing dependency on the analytics team post-handoff.

Architecture

Source → Transform → Semantic Model → Report → Users.

The SSAS Tabular model sits at the centre of the stack as the single source of truth — a pattern that maps directly to what a well-designed dbt project achieves in a modern data warehouse. The SQL transformation layer handles extraction and grain normalisation; the semantic model handles business logic; the reports handle presentation. Each layer has a single responsibility.

Outcome

~80% faster refresh. Internally owned RLS. Reports used in accreditation.

  • Dataset refresh dropped from approximately 30 minutes to approximately 6 — an ~80% reduction — achieved through star-schema re-architecture that eliminated inefficient many-to-many joins.
  • Row-level security is now managed entirely by an internal business user, trained during the engagement. No ongoing dependency on the analytics team.
  • Reports feed directly into accreditation workflows, where consistency and auditability are a requirement. The documented measure library means any stakeholder can verify what a number means.
  • Planning meetings run on current data, not figures that are hours old.

Skills & Analytics Engineering Relevance

The semantic layer is just the beginning.

SSAS Tabular is the semantic layer in this stack — the exact role a dbt project fills in a modern cloud architecture. The patterns are the same: a single place to define business logic, a shared measure library, version-controlled definitions, documented lineage. The tooling differs; the thinking doesn't.

This engagement confirmed that the biggest leverage in analytics isn't the report — it's the model underneath it. A well-designed star schema with documented measures eliminates an entire class of trust problems downstream.
