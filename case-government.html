Workforce-productivity and digital-transformation analytics for a UAE federal government HR authority — spanning 30+ entities and a national workforce of approximately 38,000 employees.

Problem

No cross-entity visibility across a 38,000-person national workforce.

A UAE federal government HR authority overseeing more than 30 entities had no unified view of workforce productivity or digital transformation progress. Each entity operated independently, reporting through different formats and at different cadences. Federal leadership had no reliable way to benchmark entities or track progress at the national level.

The challenge was both technical and structural: data lived in separate systems, access needed to be scoped by entity while still enabling federal-level roll-ups, and KPI definitions needed to be agreed and standardised across the organisation.

Context

Multi-entity hierarchy, strict access control, federal reporting requirements.

The solution needed to serve two distinct audiences: entity-level managers who could only see their own data, and federal leadership who needed cross-entity visibility and benchmarking. Any analytics platform had to enforce this boundary reliably in production.

Digital transformation KPIs were a specific requirement — tracking adoption of digital initiatives across entities alongside traditional workforce productivity metrics.

Approach

Power BI design, bilingual pages, mobile layout.

My scope on this engagement was the Power BI Report Server layer. The SQL Server data infrastructure, SSIS extraction pipeline, and SSAS Tabular semantic model were designed and built by a data engineer on the engagement.

  • Report design — designed the workforce-productivity dashboard suite across all entities: headcount and trends, productivity ratio tracking, and digital-transformation initiative progress.
  • Bilingual layouts — built full Arabic and English report versions to meet the organisation's language requirements. Arabic required right-to-left text direction and a mirrored visual composition throughout each page.
  • Mobile-optimised pages — built dedicated mobile report layouts for each dashboard, re-composing views for portrait small-screen format to support stakeholders accessing reports on tablets and phones.
  • KPI alignment — worked with the client's HR leadership to agree productivity ratio and digital-transformation KPI definitions before build. Agreed definitions embedded in DAX measure descriptions.

Architecture

Custom Sources → SSIS → SQL Server → SSAS → Power BI RS.

The full pipeline — multiple custom source systems per entity → SSIS extraction → shared SQL Server → SSAS Tabular semantic model with multi-entity RLS → Power BI Report Server → end users — was a joint delivery. The SSIS, SQL Server, and SSAS layers were built by a data engineer on the engagement; the Power BI Report Server layer (report design, bilingual Arabic/English layouts, and mobile pages) was my contribution.

Outcome

Cross-entity analytics live. Federal KPIs visible for the first time.

  • For the first time, federal leadership had a single, reliable view of workforce productivity and digital-transformation progress across all 30+ entities.
  • Entity managers could access their own dashboards with confidence that data was scoped correctly — no manual access management required.
  • Agreed KPI definitions, embedded in the semantic model, meant every number in every report meant the same thing to every stakeholder.

Skills & Analytics Engineering Relevance

Security as a first-class design concern.

Multi-entity RLS is a pattern that appears repeatedly in enterprise analytics — and getting it wrong has real consequences. The key insight is that security logic belongs in the semantic layer, not in the report. A report that controls its own access is fragile; a model that controls access reliably across every connected report is production-grade.

This is directly relevant to analytics engineering: in a dbt + BI tool stack, row-level security policies belong as documented, version-controlled artefacts in the semantic layer — not as ad-hoc filters applied per dashboard.
